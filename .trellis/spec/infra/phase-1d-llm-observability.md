# Phase 1D LLM Observability

## Scenario: Reproducible LLM Cost And Budget Records

### 1. Scope / Trigger

- Trigger: changes to LLM call fields, token pricing, cost estimation, budget
  reason codes, or PostgreSQL cost reporting.
- Applies to `packages/shared/src/llm-observability.ts`,
  `packages/llm-observability`,
  `infra/migrations/0003_llm_call_logging_cost_governance.sql`,
  `infra/verification/phase1d_llm_observability.sql`, and
  `docs/llm_observability.md`.
- Does not authorize provider invocation, prompt/completion storage, runtime
  mode transitions, billing reconciliation, or dashboard implementation.

### 3. Contracts

Cost calculation:

- Rates and stored costs use six-decimal currency values.
- Convert rates to integer micro-currency units before multiplication.
- Use `BigInt` for token/rate multiplication and round each input/output
  component to the nearest micro-unit.
- Snapshot `input_cost_per_million` and `output_cost_per_million` on every log.
- Single-call cost and configured budgets must fit PostgreSQL
  `numeric(12, 6)`.
- Ticket/day accumulated costs may exceed `numeric(12, 6)` and are accepted up
  to JavaScript's largest exactly representable micro-unit integer.

Log persistence:

- `trace_id`, `model_config_version_id`, `prompt_version_id`, and `latency_ms`
  are required.
- Trace and model config foreign keys include `tenant_id`.
- Status, rate, currency, and budget reason have no database insert defaults.
- `total_tokens` is generated by PostgreSQL; `NewLLMCallLog` omits it.
- Logs are append-only and contain no prompt content, completion content,
  secret references, or raw provider payloads.

Reporting:

- Daily views use UTC dates.
- Every aggregation groups by `cost_currency`; never add different currencies.
- Budget evaluation requires `costCurrency === budgetCurrency`.
- Zero ticket or daily budget means that limit is disabled.

Cost-cap trace propagation (AC-08):

- When projected cost exceeds the ticket or daily cap, the LLM runtime must
  return `budget_blocked` and persist a `llm_call_logs` row with
  `call_status = 'cancelled'` and the matching `budget_reason_code` (e.g.
  `ticket_budget_exceeded`) **before** returning — the blocked call is still
  logged; it is never silently swallowed.
- The ticket service must reflect that signal on the trace: after the pipeline
  runs it checks whether any `llm_call_logs` row for the trace carries an
  exceeded `budget_reason_code`, and if so sets
  `agent_traces.metadata = { "cost_cap_exceeded": true }` via a jsonb merge.
- `agent_traces.metadata` is the trace-level flag container. Do not mutate
  `agent_traces.runtime_mode` to record a cost-cap downgrade; the downgrade
  itself lives in `runtime_mode_decisions`.

### 4. Validation & Error Matrix

| Condition | Expected behavior |
|-----------|-------------------|
| Negative/non-integer token count | `invalid_integer` or `out_of_range` |
| Rate/budget has more than six decimals | `invalid_precision` |
| Single-call cost exceeds `numeric(12, 6)` | `out_of_range` |
| Accumulated cost exceeds the configured budget | exceeded reason code, not a validation error |
| Cost and budget currencies differ | `currency_mismatch` |
| Unsupported call status or budget reason | `invalid_enum` |
| Successful call has an error code | `invalid_status_error` |
| Failed/timed-out/cancelled call lacks an error code | `invalid_status_error` |
| Cross-tenant trace/config reference | PostgreSQL `foreign_key_violation` |
| Stored estimate disagrees with token/rate snapshots | PostgreSQL `check_violation` |
| Log update or delete | PostgreSQL `check_violation` |

> **Warning**: `0003` must drop the append-only trigger before compatibility
> updates when rerun. Otherwise existing rows make the migration fail before
> the trigger can be recreated.

### 5. Good/Base/Bad Cases

- Good: calculate `estimated_cost` from token counts and snapshotted rates, then
  persist the immutable config and prompt version IDs.
- Good: return `ticket_and_daily_budget_exceeded` when both projected totals
  cross enabled limits.
- Base: record a failed call with zero tokens/cost, non-negative latency, and a
  non-empty provider error code.
- Bad: infer historical cost from current provider pricing.
- Bad: let a database default silently record a successful zero-cost call.
- Bad: sum USD and EUR rows in one daily total.

### 6. Tests Required

- Unit tests must assert exact micro-unit calculation and half-micro rounding.
- Unit tests must cover all four budget reason codes, zero-disabled limits,
  currency mismatch, aggregate values above the per-row numeric limit, runtime
  enum validation, and status/error consistency.
- Unit tests must assert a budget-blocked call is logged with
  `call_status = 'cancelled'` and the exceeded `budget_reason_code` (no silent
  swallow).
- Static validation must assert package exports, migration constraints, views,
  verification cases, docs, and root scripts.
- Live PostgreSQL verification must assert generated token totals,
  tenant-consistent foreign keys, estimate consistency, append-only behavior,
  and UTC currency-grouped reporting.
- Run `npm run lint`, `npm run typecheck`, `npm run test`, migration, live
  verification, and active Trellis task validation.

### 7. Wrong vs Correct

#### Wrong

```ts
const estimatedCost =
  inputTokens * currentProviderInputRate +
  outputTokens * currentProviderOutputRate;
```

This uses floating-point multiplication, omits the per-million divisor, and
cannot reproduce historical cost after provider pricing changes.

#### Correct

```ts
const estimate = estimateLLMCallCost({
  inputTokens,
  outputTokens,
  inputCostPerMillion: snapshottedInputRate,
  outputCostPerMillion: snapshottedOutputRate,
});
```

The shared calculation uses integer micro-units, and the same rate snapshots
are persisted with the resulting estimate.
